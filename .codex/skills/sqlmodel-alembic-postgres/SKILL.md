# SQLModel Alembic Postgres

Use this skill when data-model changes span `app/models.py`, `app/schemas/`, `app/crud/`, and `app/alembic/versions/`.

## Workflow

1. Update ORM definitions in `app/models.py`.
Keep ORM models, base classes, and enums in the single-file model layout already used by the repository.

2. Update API schemas separately.
Import ORM types from `app.models` and API schemas from `app.schemas`. Do not blur the two layers.

3. Adjust CRUD queries.
Return typed tuples for paginated lists, keep business logic out of CRUD, and preserve query readability over optimization.

4. Add or review the Alembic migration.
Prefer autogenerated revisions, then inspect and fix them manually before applying.

5. Mirror the dataset model and schema comment style.
When editing `app/models.py`, related schemas, or data-heavy CRUD queries, keep the same sectioning and comment granularity as the `datasets` area instead of leaving dense structures undocumented.

## Identity Rules

- Database primary keys stay as UUID fields. Do not replace repository-wide UUID PKs with integer sequences or business codes.
- For user-facing core business entities, add a separate business display code field when the module needs non-technical IDs in UI, exports, or operations records.
- Business display codes are not a replacement for the UUID primary key; they are an additional unique field for display and search.
- Unless a requirement says otherwise, use the repository's daily sequence style such as `USR-20260528-001`, `DST-20260528-001`, `TPL-20260528-001`, `FTK-20260528-001`, or the existing training-module prefixes.
- Demo-only `Item` does not need a business display code.
- `Label` keeps its existing `code` field and should not get an extra daily-sequence display code unless a future requirement explicitly asks for it.
- When a business display code is added to a table, review whether related snapshot tables also need `{entity}_code_snapshot` fields for historical display consistency.

## Enum Rules

- Inherit from `str, Enum`.
- Provide a `.label` property.
- Store labels in a module-level mapping directly below the enum.
- Mirror enum labels in response schemas using `{field}_label` fields and one `model_validator`.

## Query Rules

- Use `select(func.count()).select_from(query.subquery())` for totals.
- Prefer `selectinload` and batch queries to avoid N+1 behavior.
- Expand midnight `end_time` values to the end of the day when filtering by date ranges.
- Keep types explicit for `dict`, `list`, and `tuple`.
- When unioning rows from heterogeneous PostgreSQL tables, especially when enum columns participate in `UNION` or `UNION ALL`, cast those columns to a shared scalar type such as `String` first instead of relying on implicit enum coercion.
- Add concise comments where query phases are not obvious, especially filters, count queries, preloading, and batch aggregation.

## Comment Rules

- Keep business-domain separator comments in `app/models.py`, for example `# ==================== Dataset ====================`.
- Add short purpose comments or docstrings for non-obvious enums, ORM classes, and relationships.
- Group schema files with section comments so request, response, list, and detail types are easy to scan.
- Keep comment density moderate: explain snapshot semantics, table responsibility, and non-obvious fields, not every field assignment.

## Migration Rules

- Name revisions after the business change, not after vague internal edits.
- Review nullability, indexes, defaults, and backfills explicitly.
- Keep changes compatible with the repository's local PostgreSQL assumptions.
- When adding a business display code to an existing table, backfill historical rows in the migration and then enforce `NOT NULL`, index, and uniqueness as needed.
